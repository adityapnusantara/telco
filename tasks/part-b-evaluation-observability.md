# Part B - Evaluation & Observability

## Before Launch - Evaluation Strategy

Describe how you would evaluate the agent before releasing to real users:

1. What test dataset you would create and how
   - Kumpulkan **100-300 sample pertanyaan customer service telco** agar evaluasi merepresentasikan kondisi nyata di produksi.
   - Sumber:
     - Historical support tickets
     - FAQ knowledge base
     - Synthetic questions generated by LLM untuk dua sumber sebelumnya, sehingga terbentuk pairing question-answer (Q&A) yang konsisten
2. What metrics you would use (e.g. answer accuracy, retrieval precision, hallucination rate)
   - Tool Use Evaluation: metric ini mengukur apakah agent berhasil memanggil tool yang tepat untuk menjawab pertanyaan, sehingga kita bisa mendeteksi apakah agent masih sering salah memanggil tool.
   - Context Precision: metric ini menilai apakah retriever menaruh chunk relevan di ranking atas dibanding chunk tidak relevan (diukur sebagai mean precision@k), sehingga kualitas konteks untuk menjawab pertanyaan lebih terjaga.
   - Context Recall: metric ini mengukur seberapa banyak informasi/dokumen relevan yang berhasil terambil dari total informasi relevan yang seharusnya ada, sehingga kita bisa mendeteksi apakah retriever masih sering melewatkan konteks penting.
   - Context Entities Recall: metric ini mengukur apakah entity penting pada jawaban referensi (mis. nama paket, biaya, batas waktu) sudah tercakup di context retrieval.
   - Answer Relevance: metric ini mengukur seberapa relevan jawaban terhadap pertanyaan user (on-topic) dan tidak memberikan informasi lain yang tidak relevan.
   - Faithfulness: metric ini mengukur apakah isi jawaban benar-benar didukung oleh context yang diambil, untuk menekan jawaban yang tidak grounded.
   - Metric Operational: Dilakukan load test menggunakan locust untuk menghitung beberapa metrics dibawah
      - API Latency (p50/p95/p99)
      - Error Rate
      - Timeout Rate
      - Throughput (RPS)
3. Whether you would use manual review, automated eval, or LLM-as-judge and why
   - Manual Review: digunakan secara targeted untuk kasus high-risk (mis. ada action yang di-execute oleh agent yang memiliki risiko tinggi)
   - LLM-as-Judge: Otomatis eval yang ada di langfuse. kita bisa add dataset untuk, kemudian melakukan experiment menggunakan API / Experiment SDK langfuse.
4. What minimum passing threshold you would require before approving a release
   - Tool Use Evaluation >= 0.95
   - Context Precision >= 0.90
   - Context Recall >= 0.90
   - Context Entities Recall >= 0.95
   - Answer Relevance >= 0.90
   - Faithfulness >= 0.90
   - Tambahan release gate:
      - Jika salah satu metric utama gagal threshold, release ditunda dan wajib perbaikan + re-run evaluasi.

## In Production - Monitoring & Observability

Once live, describe what you would monitor:

1. At least 3 specific metrics and why each one matters for this use case
- Escalation Rate: Naik tajam bisa jadi sinyal kualitas jawaban turun.
- Answer Relevance: Jika turun, bisa jadi sinyal konteks yang diambil tidak relevan.
- Faithfulness: Jika turun, bisa jadi sinyal ada jawaban yang tidak grounded.
- Metric Operational:
   - API Latency (p50/p95/p99)
   - Error Rate
   - Timeout Rate
   - Throughput (RPS)
2. What tool(s) you would use - e.g. Langfuse, LangSmith, custom logging
   - Langfuse: karena sudah include LLM-as-Judge, metrics, dan observability.
3. How you would detect and respond to a sudden drop in answer quality
   - Jalankan evaluasi berbasis sampling secara real-time pada trace production.
   - Pasang alert jika terjadi penurunan diproduction, misalkan:
      1. Faithfulness turun > 10%
      2. Answer Relevance turun > 10%
      3. Escalation Rate naik > 30%
      4. Latency, Error Rate, Timeout Rate meningkat. Dan RPS menurun.
   - Buat scheduler untuk menjalankan evaluasi secara berkala menggunakan dataset evaluasi, guna mendeteksi apakah terjadi model drift. misalkan Tool Use Evaluation, Context Precision, Context Recall, Context Entities Recall, Answer Relevance, Faithfulness menurun.

## Failure Mode Analysis

Describe how you would handle these two specific failure scenarios:

- Scenario 1: The LLM starts producing hallucinated answers - making up billing policies that do not exist in the knowledge base
   1. Cek perubahan terakhir (prompt/model/retriever/KB). Jika impact dari itu maka rollback.
   2. Buat Grounding guardrail, melalui system prompt paksa jawaban "terkait billing policies" harus berdasarkan knowledge base. Jika jawaban tidak berdasarkan knowledge base, maka escalate=true.
   3. Tampilkan jawaban aman: “Saya belum bisa memastikan dari data yang tersedia.”
   4. Tambahkan test case negatif ke dataset evaluation. kemudian jalankan evaluasi LLM-as-judge.
   5. Jika minimum threshold terpenuhi maka relase perubahan.
- Scenario 2: After a knowledge base update, the vector store starts retrieving irrelevant chunks for most queries
   1. Hentikan update knowledge base.
   2. Jika memiliki snapshot maka lakukan rollback terhadap database knowledge yang dirasa lebih stabil.
   3. Quick diagnose: 
      a. Bandingkan before / after embedding model yang digunakan, chunking strategy, metadata/filter.
      b. Dokumen baru apakah memiliki kualitas yang buruk ? sehingga menurunkan 
   4. Perbaiki yang menjadi root cause.
   5. Jika sudah diperbaiki, maka lakukan update knowledge base kembali.
   6. Jalankan evaluasi LLM-as-judge untuk memastikan bahwa evaluasi sudah kembali normal.
